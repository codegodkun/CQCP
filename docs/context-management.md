# docs/context-management.md

## 目的

本文档说明项目记忆写入规则。`AGENTS.md` 只保留强制原则和索引；详细判断、文件分工和交付输出统一放在本文档。

本项目不依赖聊天上下文保存关键决策。任何会影响后续开发、治理、范围、风险或执行顺序的信息，都必须写入项目记忆文件。

## 必须写入项目记忆的内容

以下内容必须写入项目记忆：

- 已完成任务的事实结果。
- 已确认的项目目标、业务边界、V1/V2/V3 范围变化。
- 已确认的技术决策、架构约束、接口边界、数据模型方向和治理流程。
- 已确认的审核链路、模型职责、EvidencePacket、`SYS-*` 诊断、finding 分流规则变化。
- 新发现且不会在当前任务内解决的风险、技术债、后续任务线索。
- 人工明确确认的待办顺序、优先级、限制条件和不做事项。
- 测试、验证、审查、演练或文档治理的结果。
- 影响后续 Agent 执行方式的流程规则或文档规则变化。

## 写入哪个文件

- `PROJECT_BRIEF.md`：写项目目标、用户角色、业务边界、核心流程和成功标准。只写稳定、已确认、跨阶段有效的内容。
- `CURRENT_CONTEXT.md`：写当前阶段、当前判断、正在关注的问题、待确认事项、下一步任务和当前禁止推进事项。
- `ROADMAP.md`：写 V1/V2/V3 阶段拆分、里程碑、阶段边界和阶段性验证目标。
- `docs/*.md`：写模块级长期知识。前端写 `docs/frontend.md`，后端写 `docs/backend.md`，数据库写 `docs/database.md`，AI 审核写 `docs/ai-review.md`，SAP/OA 写 `docs/sap.md`，部署写 `docs/deployment.md`。
- `docs/context-management.md`：写项目记忆写入规则、文件分工、人工确认标准和 Memory Writeback 输出格式。
- `tasks/active/TASK-xxx.md`：写当前任务范围、执行记录、完成结果、测试结果、风险与遗留问题。
- `tasks/done/TASK-xxx.md`：任务完成并归档后保留最终结果。
- `decisions/ADR-xxx.md`：写重大架构、数据、审核链路、模型职责、集成、权限或治理决策。
- `changelog/YYYY-MM.md`：写当月项目记忆变更摘要、文档治理、架构沉淀和重要任务完成记录。

## 任务文档命名与语言规范

- 任务文件名可使用英文 slug，以保持工程兼容和稳定路径，推荐格式为 `TASK-xxx-english-slug.md`。
- 面向人阅读的任务标题、目标、验收标准和交付摘要应使用中文。
- 任务正文应以中文为主；英文 slug 仅作为文件路径和工程标识，不替代中文标题和中文说明。

## 什么内容不写入

以下内容不写入长期项目记忆：

- 临时聊天推测、未验证假设、即时调试过程噪音。
- 未被用户或任务确认的建议、偏好或可能方案。
- 任务外顺手发现但尚未形成明确风险或后续任务的零散想法。
- 终端完整日志、模型完整输出、完整合同文本、敏感数据、密钥、token、endpoint 凭据。
- 可从代码或测试直接读出的低价值重复描述。
- 已废弃且不会影响后续判断的临时实现细节。
- 与本项目无关的旧 `ai-contract-review` 项目历史内容，除非通过单独任务定点提取。

## 什么内容需要人工确认

以下内容在写入时必须标记为“待确认”，除非已有明确人工确认或任务文件明确要求：

- 正式技术栈、语言、框架、目录结构。
- V1 MVP 最终范围、第一批任务包和实现顺序。
- 数据库 ERD、迁移策略、表命名、字段类型和存储策略。
- API OpenAPI 契约、错误码、认证方式、SAP/OA/采购系统集成方式。
- 管理台登录方案、权限矩阵、管理员审批职责和人员归属。
- 首批审核点清单、规则集、合同类型画像和样本集 owner。
- 本地 A30/Gemma endpoint、模型版本、部署形态、容量边界和公网模型授权边界。
- 生产、预生产、测试环境拓扑、日志指标告警、备份恢复和 secrets 管理。
- 任何来自旧项目的经验、样本、规则或反例是否迁入。

## 什么内容必须创建 ADR

以下变化必须先提出方案，并在 `decisions/` 创建或更新 ADR：

- 正式技术栈、仓库目录、模块边界或部署拓扑。
- 数据库核心模型、不可变快照、迁移策略、审计策略或存储加密策略。
- 审核链路、EvidencePacket、ReviewExecutionPlan、结果快照或状态机的重大变化。
- Gemma/A30、公网模型、prompt/schema/model version、Quality Copilot 职责边界变化。
- SAP/OA/采购系统集成契约、认证、回调/轮询、错误码和 caller policy 重大变化。
- 权限、管理员审批、发布治理、紧急变更、样本集生命周期或生产保护规则变化。
- 会改变 V1 范围、业务边界、平台职责或“不批准、不拒绝、不阻断、不修改审批流”定位的变化。

ADR 不用于记录普通任务进度、轻量文案修订、无行为变化的格式调整或临时待确认事项。

## 每次任务结束如何输出 Memory Writeback

交付摘要中必须包含 `Memory Writeback` 小节，至少说明：

- `CURRENT_CONTEXT.md`：是否更新；更新了什么；如未更新，说明原因。
- `changelog/YYYY-MM.md`：是否更新；记录了什么。
- `tasks/active/TASK-xxx.md`：是否更新；没有当前任务包时说明“本次无当前 TASK 文件”。
- `docs/*.md` / `PROJECT_BRIEF.md` / `ROADMAP.md` / `decisions/*.md`：如有更新，逐项说明；如无相关更新，可合并说明。
- `待确认`：本次新增或保留的待确认事项。
- `ADR`：本次是否需要 ADR；如不需要，说明没有触发重大决策。

推荐输出格式：

```text
Memory Writeback:
- CURRENT_CONTEXT.md: ...
- changelog/YYYY-MM.md: ...
- TASK: ...
- docs/decisions: ...
- 待确认: ...
- ADR: ...
```

## Next Task Handoff

`Next Task Handoff` 的目的是在当前任务结束时，明确告诉下一位 Agent 或新 Codex 窗口：下一步是存在可直接执行的任务，还是仅有建议、待确认事项或征求意见。该机制用于减少把临时建议误当作既定执行任务的风险。

### 何时输出可复制代码块

只有同时满足以下条件，才可以输出可一键复制的 `Next Task Handoff Prompt`，并放在独立 fenced code block 中：

- 已经存在明确的下一任务文件，例如 `tasks/active/TASK-xxx.md`。
- 任务编号明确。
- 任务目标明确。
- 可以在新 Codex 窗口中独立执行。
- 不需要用户先做方向判断。
- 不需要用户先确认重大决策。

### 何时不得输出代码块

以下情况不得输出可复制代码块：

- 只是建议下一步。
- 只是征求用户意见。
- 只是需要用户确认方向。
- 没有明确任务编号。
- 没有对应 `tasks/active/TASK-xxx.md`。
- 下一步可能涉及改变 V1 范围、技术栈、数据库、API、模型职责、SAP/OA 集成、权限或治理边界。
- 需要用户先选择多个方案之一。

这些内容只能放在普通文字中，并标为：

- `建议下一步`
- `需要人工确认`
- `可选后续任务`
- `暂不生成 Next Task Handoff Prompt`

### 可复制代码块必须包含哪些内容

当允许输出 `Next Task Handoff Prompt` 时，代码块中的提示词必须至少包含：

- 明确的下一任务文件路径，例如 `tasks/active/TASK-xxx.md`。
- “本窗口是一次性任务执行窗口，不依赖历史聊天记录作为事实来源”的约束。
- “若聊天内容与仓库文件冲突，以仓库文件为准”的约束。
- 开始前必须读取的文件：`AGENTS.md`、`CURRENT_CONTEXT.md`、对应任务文件。
- 根据任务需要按需读取 `docs/context-management.md`、相关模块文档和相关 ADR 的提示。
- 工作范围约束：只完成当前 TASK 范围，不顺手扩展，不把未确认内容写成已确认事实。
- 若当前 TASK 未明确要求，不写业务代码、不创建脚手架、不安装依赖。
- 完成后必须执行 Memory Writeback，并更新 `CURRENT_CONTEXT.md`、`changelog/当前月份.md`、当前 TASK 文件；如触发重大决策，创建或更新 ADR。
- 输出要求至少包含：完成内容、修改文件、测试或验证结果、风险与遗留问题、Memory Writeback、Next Task Handoff。

### 如果没有明确下一任务怎么办

如果当前不存在明确的下一任务文件，或者只有方向性建议、人工确认项、候选任务或待拆分工作：

- 仍然必须输出 `Next Task Handoff`。
- 但不得输出 `Next Task Handoff Prompt` 代码块。
- 只能在普通文字中说明 `建议下一步`、`需要人工确认`、`可选后续任务` 或 `暂不生成 Next Task Handoff Prompt`。

### 如何区分“执行任务”和“建议/征求意见”

可以放入 `Next Task Handoff Prompt` 的内容，必须是“执行任务”，也就是：

- 已经有任务编号。
- 已经有任务文件。
- 已经明确目标和边界。
- 不依赖用户先做方向判断或重大决策确认。

不得放入代码块的内容属于“建议/征求意见”，包括但不限于：

- 询问用户是否接受某个方向。
- 建议是否新建某个 TASK。
- 建议是否将某个 ADR 从 `Proposed` 改为 `Accepted`。
- 建议是否调整 V1 范围、技术栈、数据库、API、模型职责、SAP/OA 集成、权限或治理边界。
- 任何尚未形成明确任务包的后续工作。

## 写入原则

- 只写当前任务范围内产生的长期信息。
- 不顺手扩大范围，不把临时建议写成已确认事实。
- 不删除已有核心红线，除非有明确任务和 ADR 支撑。
- 不确定的信息统一标记为“待确认”。
- 文档更新要区分事实、判断、风险、遗留问题和建议下一步任务。
- 变更记录应说明本次未创建业务代码、未搭建脚手架、未安装依赖，除非任务本身明确相反。
