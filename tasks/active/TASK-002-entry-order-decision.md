## 背景

当前文档同时描述外部系统入口、管理台入口和评测入口。所有入口必须进入同一套审核链路，但第一轮 MVP 不宜同时完整实现所有入口。需要先决定 MVP 从管理台上传入口、外部 API 入口，还是 API-first + 简易管理台开始。

## 目标

- 决定 MVP 的第一入口顺序。
- 明确第一入口需要的最小输入、输出、权限和结果 URL 行为。
- 明确未入选入口在 MVP 中如何占位或延后。
- 为技术栈、API 契约和 scaffold 范围提供依据。

## 初步方案比较

以下为技术基线收敛阶段审查过的方案。

### 方案 A：管理台上传优先

优点：

- 内部试点和样本调试最直接。
- 可较快形成“上传 -> 审核 -> 结果页”的人工可见闭环。
- 对 SAP/OA 认证、错误码和轮询适配依赖较少。

风险：

- 若缺少 API-first 边界，后续外部系统接入可能返工。
- 容易把完整管理台能力提前带入 MVP。

### 方案 B：外部 API 优先

优点：

- 更贴近平台作为中台能力的长期定位。
- 可尽早固定 Task / Execution / Result URL 的外部契约。

风险：

- 外部 caller、认证、网络边界、轮询频率和错误码适配均待确认。
- 缺少简易人工入口时，MVP 调试和样本验证效率较低。

### 方案 C：API-first + 简易管理台

优点：

- 先固定统一审核链路和最小 API 边界。
- 简易管理台只作为内部触发和查看入口，不形成管理台专属审核路径。
- 能同时支撑技术验证、人工样本调试和后续外部 caller 接入。

风险：

- 需要严格限制简易管理台范围，避免滑向完整管理台。
- 需要尽早定义结果 URL 访问控制的临时边界。

## 已确认决策

MVP 入口顺序采用方案 C：API-first + 简易管理台。

确认理由：

- 最符合“所有入口进入同一套审核链路”的长期约束。
- 避免第一轮 MVP 变成完整管理台项目。
- 比外部 API 单独优先更适合内部样本调试和结果页验证。
- 能为后续 scaffold 提供较清晰的后端、前端和 API 边界。

该决策已记录到 `ADR-004-mvp-entry-order.md`。

## MVP 第一入口最小边界

MVP 第一入口按以下边界推进：

- 后端提供最小任务创建入口，接收合同文件、结构化字段、合同类型来源和 caller 信息。
- 简易管理台调用同一任务创建入口，不拥有独立审核链路。
- 任务创建立即返回 `taskId`、`executionId`、`status=QUEUED` 和 `resultUrl`。
- 普通结果 URL 展示 latest non-superseded execution；是否必须登录待确认。
- 外部 SAP/OA 深度 correction、完整历史查询、caller policy 管理和 contract test 延后到 Pilot 或 Production Readiness。

## 未入选入口的延后策略

- 管理台上传优先：不作为独立 MVP 入口顺序；仅保留简易管理台作为同一 API 的内部触发和查看入口。
- 外部 API 优先：不单独作为唯一 MVP 入口；最小 API 边界仍优先固定，但正式 SAP/OA/采购系统联调延后。
- 评测入口：不进入第一轮 MVP；首批样本验证可通过简易管理台或后续测试任务支撑。
