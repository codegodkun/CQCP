## 0. 任务摘要

**执行模式**：只读复查，不修改文件，不产生实现改动。

**一句话任务**：
> 对 [被复查 TASK_SPEC / 交付物] 执行 claim-based 只读复查，识别工作区状态、验证关键 claim，并输出复查报告，供 Codex 做 Review Intake Decision。

**它是什么**：
- 一种独立于执行型 `TASK_SPEC` 的专用复查类型。
- 只负责审查既有交付物、实现报告、工作区状态和 claim。
- 输出只作为 Codex 后续 Review Intake Decision 的输入。

**它不是什么**：
- 不是 implementation / test / bugfix / refactor 任务。
- 不负责修改代码、补测试、修问题、提交、归档或 push。
- 不等于最终验收通过。

**文件访问范围**：

```text
👀 允许只读参考：
  [被复查 TASK_SPEC 文件]
  [被复查交付物文件]
  [实现报告或相关输出]
  [用于验证 claim 的必要上下文文件]

⛔ 禁止写入：
  任意文件
```

## 2. 工作区规则

### 2.1 基本规则

- `readonly-review` 不要求 `git status` 为空。
- 允许在 dirty 工作区下执行复查。
- 复查者必须识别每个 dirty 文件是否属于复查范围。

### 2.2 必须执行的工作区识别

复查者必须：

1. 读取工作区状态。
2. 列出每个 dirty 文件。
3. 对每个 dirty 文件标记：
   - 属于复查范围；
   - 不属于复查范围；
   - 当前无法判断。

### 2.3 范围外 dirty 改动

对范围外 dirty 改动：

- 不得自行清理；
- 不得自行忽略；
- 只能标记为“风险项”或“必须修复项”。

是否因此阻断继续，不由复查者决定，由 Codex 在 Review Intake Decision 中判断。

## 3. 默认允许的只读命令

在没有更严格限制时，默认允许以下只读命令：

- `git status --short`
- `git status -sb`
- `git log --oneline -5`
- `git diff --stat`
- `git diff --name-status`
- `git diff`
- `git show`
- `rg` / `grep`
- 只读查看文件内容

如果具体 `TASK_SPEC` 限定了更小命令范围，以具体 `TASK_SPEC` 为准。

## 4. 禁止项

### 4.1 文件与仓库操作

不得：

- 修改文件；
- 创建文件；
- 删除文件；
- 重命名文件；
- 移动文件；
- 写入任何文件内容。

### 4.2 Git 操作

不得：

- `git add`
- `git commit`
- `git push`
- `git reset`
- `git checkout`
- `git clean`

### 4.3 测试、构建、依赖

默认不得：

- 运行测试；
- 运行构建；
- 运行 lint；
- 运行类型检查；
- 安装依赖；
- 更新依赖。

只有当当前 `TASK_SPEC` 明确授权具体命令时，才可执行指定命令。

## 5. STOP 规则

如遇以下任一情况，必须停止并回交 Codex，不得继续输出完整复查结论：

- 无法读取当前 `TASK_SPEC`
- 无法读取被复查 `TASK_SPEC`
- 无法读取目标文件
- 无法读取 `git diff` / `git status` / 必要证据材料
- 需要修改文件才能继续验证
- 需要运行未授权测试或构建才能继续验证
- 需要访问超出授权范围的材料

STOP 输出格式：

```text
[STOP: <类型> — <具体描述> — 需要 Codex 补充材料或决策]
```

不得仅凭摘要、口头汇报或不完整材料生成复查报告。

## 6. 证据标注规则

### 6.1 默认立场

- 默认怀疑，不默认信任。
- “已完成”“已验证”“未越界”“测试通过”等都只是 claim。
- claim 可以来自实现者汇报，也可以来自 Codex 汇报。

### 6.2 标注类型

每个 claim 必须标注为以下之一：

- `已验证`
- `未能验证`
- `仅可推断`
- `与汇报不符`
- `不涉及，跳过`

### 6.3 “未能验证”与“仅可推断”的区别

- `未能验证`：没有足够证据支持或反驳该 claim。
- `仅可推断`：存在间接证据，但没有直接证据，不得写成 `已验证`。

### 6.4 已验证的证据要求

`已验证` 必须附带至少一种可复核证据：

- 文件路径 + 行号；
- `git diff` / `git show` / `git status` 输出；
- 测试断言位置；
- 其他可复核的只读证据。

### 6.5 测试证据边界

测试通过不等于细节为真。

测试只能证明：

- 被断言覆盖到的内容；
- 在该命令、该输入、该断言下的结果。

## 7. 输出格式

复查报告最少包含以下部分：

### 7.1 已通过项

列出已验证且当前可接受的事项。

### 7.2 风险项

列出：

- 范围外 dirty 改动；
- 证据不足但存在疑点的事项；
- 需由 Codex 关注的风险。

### 7.3 必须修复项

列出：

- 与汇报不符的事项；
- 明确越界的事项；
- 明确违反禁止项的事项；
- 会影响 Codex 是否继续接纳的事项。

### 7.4 可后续登记项

列出本轮不在当前范围内，但值得后续登记为任务或治理事项的内容。

### 7.5 是否建议继续

必须给出以下之一：

- 建议继续
- 有条件继续
- 不建议继续

### 7.6 是否建议清理或分离 dirty 工作区

必须给出以下之一：

- 建议无需处理
- 建议清理
- 建议分离
- 建议由 Codex 决定

### 7.7 只读合规声明

必须声明：

- 未修改文件；
- 未创建文件；
- 未删除文件；
- 未执行 Git 写操作；
- 未运行未授权测试/构建；
- 未安装依赖。

---

## 8. Review Intake Decision 边界

- 复查报告只是 Codex 的决策输入。
- 不等于最终验收通过。
- 复查者不得自行决定提交、归档、push 或接纳结果。
- Codex 必须单独输出 Review Intake Decision。

---

## 9. 复查报告（执行者完成后填写）

### 9.1 工作区状态

- `git status --short` / `git status -sb`
- dirty 文件清单
- 每个 dirty 文件是否属于复查范围

### 9.2 Claim 复查结果

按 claim 列出：

- claim 内容
- 标注类型
- 证据

### 9.3 结构化结论

- 已通过项
- 风险项
- 必须修复项
- 可后续登记项
- 是否建议继续
- 是否建议清理或分离 dirty 工作区
- 只读合规声明

### 9.4 STOP 记录

如有 STOP，记录原文与触发原因；无则写“无”。
